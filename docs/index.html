The Problem

If you've used an AI coding assistant for more than 20 minutes on the same codebase, you've seen this: the model "remembers" validateUser(id) when the function was renamed to verifyToken(token, strict) fifteen turns ago. It confidently suggests calling a method with two parameters when the signature was updated to take four. It hallucinates a function that you deleted.

This isn't a bug in any specific model. It's context compression collapse, as the conversation grows, the model compresses earlier context into summaries, and those summaries lose structural precision. The model doesn't "forget" your code. It remembers a lossy approximation that diverges further from reality with every turn.

We wanted to measure exactly how bad this gets, and whether a simple fix, anchoring the model's memory with deterministic parser output, actually works.

The Benchmark

We built a 50-turn evolving Python codebase that starts with 5 simple functions and grows to 17 symbols across 2 classes through realistic mutations: renames, parameter additions, type changes, method extractions, class renames, async conversions, and deletions.

At each turn, we asked Claude to recall every function signature. We compared two modes: raw LLM recall (Claude maintaining its own running summary) vs OMP-anchored recall (Claude receiving Tree-sitter-extracted signatures as factual anchors).

The Token Insight

Drift is a function of compressed context size, not turn count. A turn that renames one parameter barely grows the summary. A turn that extracts 5 methods into a class can double it.

Look at the Claude results: turns 1-15 hold at 100% with ~388-591 context tokens. At turn 20, the summary nearly triples to ~1,169 tokens (that's when TokenService and PermissionService classes were extracted), and accuracy drops to 66.7%. It occasionally recovers, turn 30 hits 92.3% when a function deletion shrank the summary, but the trend is clear. At ~1,000+ summary tokens, the model's compressed context starts losing structural precision.

This matters because it means the failure mode isn't "the model is bad at coding." It's that LLM summarization is lossy compression, and the loss compounds. Every summary of a summary loses a little more.

The Fix: Parser-Anchored Memory

The idea is simple: don't ask the LLM to remember structural code facts. A deterministic parser already knows them perfectly.

Symbolic Track (Tree-sitter)

  • Function signatures
  • Parameter names, types, defaults
  • Return types
  • Class/method relationships
  • AST hashes for staleness
  • Import dependencies

Semantic Track (LLM)

  • Why was this change made?
  • Architectural preferences
  • Implicit constraints
  • User frustrations / biases
  • Unresolved ambiguity
  • Intent behind the code

The LLM is forbidden from rewriting the Symbolic Track. It can only populate the Semantic Track. The parser remains a rigid, unchangeable anchor of truth. When the agent needs to recall a function signature, it gets it from Tree-sitter, not from a summary of a summary of a summary.

flowchart TB
    CodeChange["Code Change"] --> Parser["Tree-sitter Parser"]
    CodeChange --> Observer["Observer LLM"]

    Parser --> Symbolic["Symbolic Layer
    (DETERMINISTIC)

    AST hashes, imports"]
    Observer --> Semantic["Semantic Layer
    (PROBABILISTIC)

    preferences"]

    Symbolic --> Reconcile["Dual-Track Memory"]
    Semantic --> Reconcile

    Reconcile --> Storage["SQLite / Any Backend"]

This is what the OMP-anchored column in the benchmark does: at each turn, Tree-sitter extracts the current function signatures and injects them as FACTUAL ANCHORS into the prompt. The model is told not to contradict them. Result: 100% accuracy at every turn, regardless of conversation length or context size.

What This Saves

Beyond accuracy, parser-anchored memory saves tokens. Instead of re-sending the full source code every turn so the model can "re-learn" what it forgot, you send a compact structured JSON of signatures (~200 tokens for 17 functions). The model gets perfect recall without paying for a full context refresh.

In our benchmark, the baseline approach consumed the full source code at every turn. With OMP, you could skip that entirely for structural facts, the parser already has them. The LLM's token budget is freed up for what it's actually good at: understanding intent, suggesting patterns, and reasoning about architecture.

# Clone and install
git clone https://github.com/open-mem/omp.git
cd omp

# Run the deterministic benchmark (no API key needed)
python -m benchmarks.run

# JSON output for your own analysis
python -m benchmarks.run --json --output results.json

The simulated mode uses a seeded RNG, you'll get the exact same table every time. The full methodology is documented in the repo.

How It Works

from omp import extract_from_source

result = extract_from_source("""
class AuthService:
    async def validate_token(self, token: str, strict: bool = True) -> User | None:
        return jwt.decode(token, self.secret, algorithms=["HS256"])
""", "python")

for fn in result.functions:
    print(fn.qualified_name)  # AuthService.validate_token
    print(fn.parameters)      # [Parameter(name='token', type='str'), ...]
    print(fn.return_type)     # User | None
    print(fn.ast_hash)        # a3f8... (changes when code changes)

Supports Python, TypeScript, JavaScript, and Go. The extraction is fast (<1ms per file), deterministic, and works offline.
